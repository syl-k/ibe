# 統合型ブラウザ・ターミナル 要件定義

> 本ドキュメントは要件の合意内容を記録するもの。設計・実装が進むにつれて更新する。
> 最終更新: 2026-06-30

---

## 1. 目的・コンセプト

1つのアプリを開けば、**ブラウザ機能とターミナル（コマンド実行）を同一アプリ内で並行利用**できる統合環境を提供する。

既存ブラウザ（Chrome / Edge）との最大の差別化点は **分割ビューに制限を設けない** こと。Chrome 等が基本2分割なのに対し、本アプリは画面を任意の数・任意の構成に分割できる。

### 想定ユースケース
- 幅広ディスプレイで4分割し、左2画面にブラウザ、右2画面にターミナルを表示。
- ドキュメント（ブラウザ）を見ながら、隣のペインでコマンドを実行する開発フロー。
- タブで複数のレイアウトを切り替えて並行作業。

---

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **ペイン (Pane)** | 分割された個々の領域。ブラウザまたはターミナルのいずれかを表示する最小単位。 |
| **分割 (Split)** | ペインを上下または左右に分けること。再帰的にネスト可能。 |
| **タブ (Tab)** | 1つの分割レイアウト全体をまとめた単位。タブを切り替えると別のレイアウトに切り替わる。 |
| **ターミナルセッション** | 1ペイン内で起動している shell プロセス（pty）。1ペイン内で複数持てる。 |

## 3. スコープ

### 3.1 対象プラットフォーム
- **当面は macOS のみ**（Apple Silicon / Intel）。
- 将来的な Windows / Linux 対応は設計上の足かせにならない範囲で意識するが、要件外。

### 3.2 MVP（最初の動く版）の優先順位
**分割レイアウト基盤を最優先**で構築する。中身（ブラウザ / ターミナル）は最低限の動作でよい。

MVP に含むもの:
1. 再帰分割型レイアウト（上下・左右に無制限分割、ドラッグでリサイズ）
2. ペインへのブラウザ / ターミナル割り当て
3. 複数タブ（レイアウト単位の切り替え）
4. 最低限のブラウザ表示（URL入力・ページ表示・戻る/進む）
5. 最低限のターミナル（ログイン shell の起動と入出力）
6. レイアウトとタブの状態復元

MVP に含まないもの（将来）:
- Chrome 拡張サポート
- 履歴 / ブックマーク UI
- 複数プロファイル
- ターミナルのリッチ表示（画像等）

## 4. 技術スタック

| 領域 | 採用技術 | 備考 |
|------|----------|------|
| アプリ基盤 | **Electron** (Chromium + Node.js) | ブラウザ機能が最も充実。クロスプラットフォーム余地あり。 |
| 言語 | TypeScript | レンダラ / メイン両方。 |
| ブラウザ描画 | `BrowserView` / `WebContentsView`（推奨）または `<webview>` | 後述「アーキテクチャ」参照。Web コンテンツはレンダラ DOM とは分離した WebContents で表示。 |
| ターミナル描画 | **xterm.js** | エミュレータ UI。 |
| ターミナルプロセス | **node-pty** | ログイン shell（zsh 等）を pty で起動。 |
| 配布形態 | **OSS 公開** | ライセンス・コントリビューションガイドは別途策定。 |

UI フレームワーク（React / Svelte / Solid 等）は未決。レイアウトツリーの状態管理が要となるため、リアクティブに扱いやすいものを実装着手前に決定する（[9. 未決事項](#9-未決事項)）。

## 5. 機能要件

### 5.1 分割レイアウト（差別化の中核）

- **モデル: 再帰分割型**（tmux / VSCode 風）。
  - レイアウトは二分木（または n 分木）で表現。各ノードは「分割（方向 + 子）」または「葉（ペイン）」。
  - 任意のペインを上下 / 左右にさらに分割でき、ネスト深さに制限を設けない。
- ペインの境界をドラッグして**リサイズ**できる。
- ペインを**閉じる**と、兄弟ペインが空間を引き継ぐ（ツリーを再構成）。
- ペインの**フォーカス移動**（マウス、できればキーボードでも）。
- 各ペインは**ブラウザ / ターミナルを自由に割り当て・切り替え**可能。
- **タブ**: レイアウトツリー全体を1タブとして保持し、複数タブを切り替えられる。

### 5.2 ブラウザ機能

MVP:
- URL 入力・ページ表示・戻る / 進む・リロード・停止。
- ペイン内のアドレスバー。
- ページタイトル / ファビコン表示。

MVP 後:
- 履歴 / ブックマーク（保存・一覧・検索）。
- **Chrome 拡張のサポート**（※制約あり → [8. 制約・リスク](#8-制約リスク)）。
- 複数プロファイル / セッション分離（将来候補）。

### 5.3 ターミナル機能

MVP:
- **ユーザーのログイン shell（zsh 等）をそのまま起動**し、`.zshrc` 等の設定を尊重する（node-pty）。
- 標準的なエミュレータ機能（色・リサイズ・コピー&ペースト・スクロール）。
- **1ペイン内で複数ターミナルセッション（ターミナル内タブ）** を持てる。

MVP 後:
- リッチ表示（画像・URL リンククリック等、iTerm2 相当）。
- shell 連携（カレントディレクトリ検知など）。

### 5.4 状態復元

- アプリ再起動時に **レイアウト構成・タブ・各ブラウザペインの URL を復元**する。
- **ターミナルのプロセス自体は復元しない**（ペインの枠は復元するが shell は再起動）。

---

## 6. 非機能要件

- **パフォーマンス**: 多ペイン（例: 4〜8ペイン同時）でも操作が引っかからないこと。各 WebContents / pty のリソースを監視できる設計が望ましい。
- **セキュリティ**（OSS + 信頼できない Web を表示するため重要）:
  - レンダラの `nodeIntegration` は無効、`contextIsolation` は有効を原則とする。
  - Web コンテンツは UI レンダラとは別の WebContents（`BrowserView`/`WebContentsView`/`<webview>`）に隔離する。
  - Node / pty へのアクセスは preload + IPC 経由の最小権限に限定する。
- **保守性**: レイアウトツリー・ブラウザ・ターミナルをモジュールとして疎結合に保つ。
- **OSS 品質**: README / ライセンス / コントリビューションガイド / 基本的なテストを整備する（リリースに向けて段階的に）。

## 7. アーキテクチャ概要（案）

```
┌─────────────────────────────────────────────┐
│ Electron Main プロセス                        │
│  - ウィンドウ管理 / アプリライフサイクル        │
│  - node-pty（shell プロセス生成・I/O）         │
│  - 状態の永続化（レイアウト/タブ/URL）          │
└───────────────▲─────────────────────────────┘
                │ IPC（preload 経由・最小権限）
┌───────────────┴─────────────────────────────┐
│ UI レンダラ（シェル UI）                        │
│  - レイアウトツリーの状態管理                   │
│  - 分割 / リサイズ / タブ UI                   │
│  - xterm.js（ターミナル描画、I/O は IPC）       │
└───────────────┬─────────────────────────────┘
                │ 各ペインに割り当て
        ┌───────┴────────┐
        ▼                ▼
  ブラウザ用 WebContents   （ターミナルは xterm.js を
  （BrowserView 等で隔離）   レンダラ内に描画）
```

検討メモ:
- ブラウザ表示は `<webview>` より **`WebContentsView`（新 API）** が推奨方向。ただし任意座標・任意サイズへの重ね合わせ（多ペイン）と、レンダラ DOM ベースのレイアウトとの座標同期が論点になる。実装着手前にプロトタイプで検証する。
- 各ターミナルセッションは Main 側で1 pty を持ち、レンダラの xterm.js インスタンスと IPC で結ぶ。

## 8. 制約・リスク

1. **Chrome 拡張サポートの制約（重要）**
   Electron の拡張サポートは限定的。`session.loadExtension` で一部拡張は読み込めるが、Chrome Web Store の **Manifest V3 拡張をフル機能でそのまま動かせるとは限らない**。本アプリの「Chrome 拡張サポート」は MVP 後の課題とし、対応範囲を実機検証のうえ確定する。

2. **多ペインでの WebContents 重ね合わせ・座標同期**
   `BrowserView`/`WebContentsView` はネイティブビューとしてレンダラ DOM の上に重なるため、分割レイアウトの境界やリサイズとピクセル単位で同期させる必要がある。MVP の技術的な最大の山場。

3. **リソース消費**
   各ブラウザペインが独立した WebContents を持つため、ペイン数に比例してメモリを消費する。多ペイン時の挙動を早期に計測する。

4. **セキュリティ**
   信頼できない Web を表示する以上、レンダラの権限設計（context isolation / preload）を最初から厳格にする。後付けは困難。
